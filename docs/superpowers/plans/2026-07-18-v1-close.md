# v1 Close Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close v1 as a documentation-only milestone: record the win criterion as NOT MET with its ceiling arithmetic, amend the v1 design doc, and correct the repo front door.

**Architecture:** Three documentation artifacts in dependency order — a v1-design closing amendment that points at the already-committed verdict spec, a README refresh that states the honest outcome and fixes two stale facts, and a closing verification record appended to the verdict spec. No code changes; the milestone's own verification asserts the code diff is empty.

**Tech Stack:** Markdown; `git`; `mise` tasks (`test`, `lint`); `cargo test` for the three differential suites. No new dependencies, no benchmarks, no metal provisioning.

## Global Constraints

These apply to every task. They come from the spec (`docs/superpowers/specs/2026-07-18-v1-close-design.md` §Scope Decisions and §Verification) and from `AGENTS.md`.

- **Branch:** `v1-close`. It already exists and already carries commit `7267c13` (the verdict spec). Do not create a new branch and do not rebase.
- **No code changes.** `git diff main -- crates/ cli/ fuzz/ scripts/` must be empty at every commit in this plan. Only `docs/` and `README.md` may change.
- **No new benchmark data.** Do not run `mise run bench`, `mise run verify-quiet-hw`, or any `mise run metal-*` task. Metal budget for this milestone is zero.
- **Recorded data points are append-only.** Never edit an existing figure, table row, or Amendments entry in any spec under `docs/superpowers/specs/`. Corrections are appended as errata; this milestone should need none.
- **Every figure is quoted, never recomputed.** Each number written into any document must be copied verbatim from the milestone spec that recorded it. Do not re-derive, re-round, or restate a ratio from memory.
- **Governing ratio basis:** llama.cpp **best-of-builds**, **full-thread**, from the `--json` protocol run. Never cite the plain-build ratio, the `t=1` diagnostic row, or a gate A/B table as the v1 result.
- **Standing verdict figures** (the only ratios any document in this plan may present as *the* v1 result): 6336Y 16c = **pp 0.83x | tg 0.96x**; E-2388G 8c = **pp 0.69x | tg 0.86x**.
- **LLVM version of record:** **22.1.8** (`devenv.nix` uses `pkgs.llvmPackages_22`, `LLVM_SYS_221_PREFIX`). The string "LLVM 18" is stale everywhere it appears.
- **Commit style:** follow the repo's existing `specs:` / `docs:` prefixes as seen in `git log --oneline`.

## Notes for the implementer

- **This milestone deliberately produces no working software.** The usual TDD cycle does not apply — the "tests" here are the grep and `git diff` assertions in each task, and they are as binding as unit tests. Do not skip them because the change is "just docs."
- **The strongest temptation is to soften the verdict.** tg 0.96x is close to parity, and the 16c session even shows a 1.00x row against one llama build. Neither is a win under the criterion ("vs llama at its best"). If you find yourself writing "essentially matched" or "within noise of beating", stop and use the recorded number instead.
- **If a figure looks wrong, the owning milestone spec wins.** Never edit a recorded data point to resolve a discrepancy; correct the citing document, or append an erratum if the source itself is wrong.
